# Plan PRów (kolejność rekomendowana)

Poniższa lista opisuje rekomendowaną kolejność Pull Requestów dla rozbudowy labu o obserwowalność (Prometheus/Grafana, ELK) oraz warstwę sterowania (Lab API + React).

| PR | Tytuł (branch) | Cel | Co dokładnie wchodzi | DoD (Definition of Done) |
|---:|---|---|---|---|
| 1 | **Observability network & baseline plumbing** (`feature/obs-net`) | Przygotować fundament pod monitoring | `docker-compose`/`compose.override` dodaje `observability_net`; wspólne `labels`, `container_name`, sensowne `healthcheck`; porty tylko gdzie trzeba | `docker compose up` działa; kontenery w osobnej sieci; opis w `docs/observability.md` |
| 2 | **Prometheus + Grafana + cAdvisor** (`feature/prom-graf`) | Metryki host/kontenery + dashboardy | Serwisy: `prometheus`, `grafana`, `cadvisor`; konfiguracja scrape dla cAdvisor; w Grafanie folder “DNS Lab” + 1 dashboard (CPU/RAM/NET per container) | Grafana pokazuje dane dla: `dns_resolver`, `dns_resolver_plain`, `dns_lab_api`, `dns_react_ui`; eksport dashboardów do repo |
| 3 | **Unbound metrics (real DNS stats)** (`feature/unbound-metrics`) | Metryki stricte DNS: QPS, cache, SERVFAIL, NXDOMAIN | Dodajesz **unbound exporter** (najprościej jako osobny kontener/sidecar), który czyta `unbound-control stats_noreset` i wystawia `/metrics`; osobne joby dla `dns_resolver` i `dns_resolver_plain` | Prometheus widzi metryki Unbound; Grafana dashboard “Resolver DNS stats”; potrafisz pokazać różnicę ON/OFF (aggressive) po kliknięciu demo |
| 4 | **BIND authoritative metrics** (`feature/bind-metrics`) | Metryki BIND (authoritative) | BIND: `rndc stats` do pliku + exporter/parser → `/metrics` (albo gotowy bind exporter); metryki typu: queries, response codes | Grafana dashboard “Authoritative stats”; metryki odczytywalne bez wchodzenia do kontenera |
| 5 | **Structured logging baseline (pre-ELK)** (`feature/log-format`) | Ujednolicić logi zanim pójdą do ELK | Ustaw logowanie Unbound/BIND na czytelny format (min. stałe prefixy; opcjonalnie JSON jeśli realne); wolumeny `/var/log/...` albo stdout | Masz przykładowe logi dla DNSSEC/SERVFAIL/NSEC w jednym miejscu + opis filtrów |
| 6 | **ELK minimal stack (single-node)** (`feature/elk-min`) | Logi w Kibanie | Serwisy: `elasticsearch` (ograniczony RAM), `kibana`, `filebeat` (docker input lub log files input), pipeline (opcjonalnie `logstash`) | W Kibanie widzisz logi z resolver/auth; index pattern działa; masz zapisane Kibana saved objects (export) |
| 7 | **Correlation: link metrics↔logs for demos** (`feature/obs-correlation`) | Połączyć “co widać na wykresach” z logami | W Grafanie linki do Kibany (np. “View logs for this container/time range”); w Kibanie saved searches dla: DNSSEC validation failures, NXDOMAIN flood | 1 klik z Grafany prowadzi do logów; saved searches działają |
| 8 | **Lab API: Nodes registry (without docker.sock)** (`feature/lab-api-agents-v1`) | “Ładniej produkcyjnie”: agent API dla resolver/auth | Dodajesz mały agent tylko na resolver/auth: `/status`, `/config` (allowlista), `/stats` (krótkie), opcjonalnie `/metrics` (jeśli chcesz); `dns_lab_api` agreguje to w `GET /nodes` | React/CLI dostaje jedną listę węzłów z rolami + IP + wersje; brak docker.sock |
| 9 | **React UI: Overview + Nodes + Quick links** (`feature/react-lab-overview`) | Panel sterujący labem | React strony: “Overview”, “Nodes”, “Configs (read-only)”; linki do Grafany/Kibany; statusy | UI pokazuje wszystkie kontenery logicznie (role), nie tylko nazwy; działa na Twoich nazwach (`dns_resolver`, `dns_signer`, …) |
| 10 | **Run Demo + Proof (aggressive NSEC)** (`feature/demo-proof`) | Powtarzalny scenariusz + dowody | `POST /demo/aggressive-nsec` w `lab_api`: odpala serię zapytań (np. 50–200 NXDOMAIN), zbiera: `dig` output, różnicę `unbound stats`, opcjonalnie pcap z `dns_capture_resolver`; UI pokazuje wynik i pozwala pobrać artifact (JSON/ZIP) | Masz “ON vs OFF” porównanie; w metrykach widać spadek upstream queries; artifact zapisany |
| 11 | **Security hardening for observability** (`feature/obs-security`) | Żeby to wyglądało “produkcyjnie” | Ograniczenia sieci (observability tylko z mgmt), hasła do Grafany/Kibany, blokada publicznych portów, `mem_limit` dla ES | Stack działa bez wystawiania ES na świat; minimalne creds; opis w docs |
| 12 | **Thesis-ready docs + diagrams** (`docs/observability-chapter`) | Materiał do rozdziału | Diagram architektury (control-plane vs observability-plane), opis przepływów danych, screenshoty dashboardów i saved searches | Masz gotowy opis do “Część badawcza/testy” + “Wnioski” |
